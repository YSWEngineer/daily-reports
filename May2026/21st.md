## 💡 本日の学び・気付き
### データベースとは？
> 情報を整理して保存し、効率的に検索・管理・共有するための仕組みのこと

---

### PostgreSQLとは？
#### 読み方
**ポストグレスキューエル** もしくは **ポストグレス**。
「ポスグレ」と略されることもある。
私はポストグレスキューエル界隈。
#### 何をしてくれるの？
一言で言うと、**データを整理して保存・管理してくれるソフトウェア**。
#### 何故PostgreSQLなのか？
データベースのソフトウェアはいくつか種類がある。

|名前   |特徴   |
|---|---|
|PostgreSQL   |高性能・信頼性が高い・無料   |
|MySQL   |Web開発でよく使われる・無料   |
|SQLite   |超軽量・小規模向け   |
|Oracle   |企業向け・有料   |

FBC で PostgreSQL を採用しているのは、Rails の現場でよく使われていて、かつ無料で高機能だから。

---

### リレーショナルデータベースとは？
一言で言うと、**表と表を関連付けて管理するデータベース**。
#### まず「表」だけのデータベースをイメージする
たとえば注文管理を考えてみる。
表が1つだけだとこうなる。
|注文id   |顧客名   |顧客の住所   |商品名   |金額   |
|---|---|---|---|---|
|1   |田中さん   |東京都〇〇   |りんご   |200   |
|2   |田中さん   |東京都〇〇   |バナナ   |150   |
|3   |鈴木さん   |大阪府〇〇   |りんご   |200   |

↑ よく見てみると、田中さんの住所が2回出てきている。
これには問題がある。
- もし、田中さんが引っ越したら**2行全部を直さないといけなくなる**
- データが増えるほど**同じ情報が何度も重複してしまう**
#### リレーショナルデータベースの解決策
表を**役割ごとに分けて関連づける**。
##### 顧客テーブル
|顧客id   |顧客名   |住所   |
|---|---|---|
|1   |田中さん   |東京都〇〇   |
|2   |鈴木さん   |大阪府〇〇   |

##### 注文テーブル
|注文id   |顧客id   |商品名   |金額   |
|---|---|---|---|
|1   |1   |りんご   |200   |
|2   |1   |バナナ   |150   |
|3   |2   |りんご   |200   |

↑ 注文テーブルの「顧客id」が顧客テーブルの「顧客id」と**リンク**（関連）している。
もし、田中さんが引っ越しても顧客テーブルの1行を直すだけでOK！

#### 「リレーショナル」の意味
**Relational = 関連のある**、という意味。
表と表を関連づけて管理するから**リレーショナルデータベース**と呼ぶ。
略して**RDB**とも言う。

---

### PostgreSQLとの関係
PostgreSQL はこのリレーショナルデータベースの仕組みを持ったソフトウェアである。
正式には**RDBMS**（リレーショナルデータベース管理システム）と呼ばれる。

---

### マスタとトランザクション
#### 2種類のテーブル
テーブルには大きく2種類ある。
##### マスタテーブル
他のテーブルから参照されることが多く、レコードの追加がそれほど頻繁でないテーブル。
先程の例だと「顧客テーブル」「商品テーブル」がこれにあたる。
一度登録したらそう変わらないデータ、というイメージ。
##### トランザクションテーブル
購入のたびにレコードが追加されるような、頻繁に更新されるテーブル。
先程の例だと「注文テーブル」がこれにあたる。
日々どんどん増えていくデータ、というイメージ。

#### 「設計が破綻している」とはどういうこと？
先程のテーブルをもう一度見てみる。

|注文id   |顧客id   |商品名   |金額   |
|---|---|---|---|
|1   |1   |りんご   |200   |

この設計だと**1回の注文で1種類の商品しか買えない**。
「りんごとバナナを一緒に買いたい」となったとき、レコードを2つ作ると今度は「この注文で合計いくら支払ったのか」「クーポンで割引があったのか」「一部だけ返品したい場合は？」といった情報が管理できなくなってしまう。
これを「**設計の破綻**」という。
最初の設計が甘いと、後から対応できない問題が次々と出てくる。

#### 何故設計が大事なのか？
データベースの設計は**後から変更することがとても難しい**という特徴がある。
アプリケーションのコードは後から修正できるが、データベースの構造を途中で変えると、すでに保存されているデータへの影響が大きく、非常に大変な作業になってしまう。
だからこそ**最初の設計を慎重に行う**ことが求められる。

#### 良い設計のチェックポイント

|チェック項目   |意味   |
|---|---|
|データに無駄な重複がないか   |同じ情報が複数の場所に書かれていないか   |
|関連するデータがまとまっているか   |1つのテーブルに関係ないデータが混ざっていないか   |
|不要なデータを保存しようとしていないか   |使わないデータまで入れていないか   |
|関連付けを正しく行えているか   |テーブル同士のつながりが適切か   |
|拡張に対応可能か   |機能追加や要件変更に柔軟に対応できるか   |

### SQLの基本操作
#### SELECT：データを取り出す
データベースから**データを検索・取得する**命令。
最もよく使う SQL。
```sql
SELECT 名前, 年齢 FROM 顧客テーブル;
```
「顧客テーブルから、名前と年齢を取り出す」という意味。

全列取り出したい場合は`*`を使う。
```sql
SELECT * FROM 顧客テーブル;
```

条件を絞りたい場合は`WHERE`を使う。
```sql
SELECT * FROM 顧客テーブル WHERE 年齢 = 30;
```

#### INSERT：データを追加する
テーブルに**新しいレコードを追加する**命令。
```sql
INSERT INTO 顧客テーブル(名前, 年齢) VALUES ('田中さん', 30);
```
「顧客テーブルに、名前が田中さん・年齢が30のレコードを追加する」という意味。

#### UPDATE：データを変更する
すでにある**レコードの内容を書き換える**命令。
```sql
UPDATE 顧客テーブル SET 年齢 = 31 WHERE 名前 = '田中さん';
```
「顧客テーブルの、田中さんの年齢を31に変更する」という意味。

#### DELETE：データを削除する
テーブルから**レコードを削除する**命令。
```sql
DELETE FROM 顧客テーブル WHERE 名前 = '田中さん';
```
「顧客テーブルから、田中さんのレコードを削除する」という意味。

#### トランザクション：コミットとロールバック
複数の操作を**ひとまとまりにして扱う**仕組み。

##### 銀行振込を例にイメージしてみる
1. Aさんの口座から1万円を引く
2. Bさんの口座から1万円を足す

この2つはセットで成功しなければならない。
1 だけ成功して 2 が失敗すると、お金が消えてしまう。
これを防ぐのが**トランザクション**。

```sql
BEGIN;  -- トランザクション開始
UPDATE 口座 SET 残高 = 残高 - 10000 WHERE 名前 = 'Aさん';
UPDATE 口座 SET 残高 = 残高 + 10000 WHERE 名前 = 'Bさん';
COMMIT;  -- 問題なければ確定
```
**COMMIT**（コミット）：すべての操作が成功したため**確定する**命令。

```sql
BEGIN;
UPDATE 口座 SET 残高 = 残高 - 10000 WHERE 名前 = 'Aさん';
UPDATE 口座 SET 残高 = 残高 + 10000 WHERE 名前 = 'Bさん';
ROLLBACK;  -- 何か問題があれば取り消し
```
**ROLLBACK**（ロールバック）：途中で問題が起きたため**BEGINの前の状態に戻す**命令。

#### まとめると......
|命令   |意味   |イメージ   |
|---|---|---|
|BEGIN   |トランザクション開始   |「ここから一連の作業を始める」   |
|COMMIT   |確定   |「全部うまくいったため保存する」   |
|ROLLBACK   |取り消し   |「問題が起きたため全部なかったことにする」   |

#### 5つの命令の全体像
|命令   |役割   |補足   |
|---|---|---|
|SELECT   |取得   |データを見る   |
|INSERT   |追加   |データを入れる   |
|UPDATE   |更新   |データを書き換える   |
|DELETE   |削除   |データを消す   |
|トランザクション   |管理   |操作をまとめて安全に行う   |
