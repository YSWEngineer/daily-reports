## 💡 本日の学び・気付き
### HTTPとは？
- Hypertext Transfer Protocol の略。
- Webページを送受信するためのプロトコル。
    - プロトコル（Protocol）：コンピュータ同士が「どうやり取りするか」を決めたお約束（ルール）のこと。
    人間の世界にも「日本語で話す」「英語で話す」「敬語を使う」などのルールがあるように、コンピュータも、通信するときはあるルールがないと意思疎通ができない。その「通信のルール」がプロトコル。

### URIとは？
- Uniform Resource Identifier の略。
- Web上の「場所」を一意に示すための名前（住所）。
    - 家の住所がないと郵便物が届かないように、Web上のリソース（HTML、画像、API……）にも住所が必要。それが URI。

### URI と URL の違い
- URI：Web上の資源を識別するための「名前」の総称。
- URL：その中でも場所（Location）を示すもの。

### HTTPのバージョンの変遷
- HTTP/0.9 → HTTP/1.0 → HTTP/1.1 → HTTP/2 → HTTP/3
- HTTP/3

    > - Google が開発していた TCP に代わるトランスポート層プロトコル QUIC を使用します。
    > - QUIC は TCP の代わりに UDP 上に独自のトランスポート層を持ち、TLS 1.3 に相当するセキュリティ機能も取り込んでいます。
    > - ポート番号は UDP 443番を使用します。
    > - HTTPS 通信、証明書は必須となります。
    > - 仕様の古い TCP の代わりに独自の高速プロトコルを使用することで転送効率や速度を向上させています。
    > - IE を除く大半のブラウザでサポート済です。
    > - 現時点 (2024年12月) の HTTP/3 使用率は 33.2% だそう

### TCPとは？
- 「確実に届ける」ことを最優先する宅配便。
- 特徴
    - データを「順番通り」「欠けることなく」「確実に」届ける。
    - 途中で失敗したら再送する。

### UDPとは？
- 「スピード優先の投げっぱなし配送」
- 特徴
    - 届くかどうか保証しない。
    - 順番が入れ替わることもある。
- 使われる場面
    - オンラインゲーム
    - Zoom・ビデオ通話
    - ライブ配信
    - DNS

### QUICとは？
- UDP の速さ + TCP の信頼性 + TLS の安全性 を全部載せした次世代プロトコル。
- 現代のブラウザが使う HTTP/3上で動く。
- HTTP をもっと速く、もっと安全にするために Google が作った新しい仕組み。

### TLS とは？
- Web通信を盗み見されないように暗号化する技術。

### HTTP のサンプル
- ブラウザで Webページを開く際、ブラウザはサーバに要求メッセージを送信する。
```
# HTTP
GET / HTTP/1.1
Accept: image/gif, image/jpeg, */*
Accept-Language: ja
Accept-Encoding: gzip, deflate
User-Agent: Mozilla/4.0 (Compatible; MSIE 6.0; Windows NT 5.1;)
Host: www.xxx.zzz
Connection: Keep-Alive
```
↑これに対してサーバは下記のような応答メッセージを返す。
```
# HTTP
HTTP/1.1 200 OK
Date: Sun, 11 Jan 2004 16:06:23 GMT
Server: Apache/1.3.22 (Unix) (Red-Hat/Linux)
Last-Modified: Sun, 07 Dec 2003 12:34:18 GMT
ETag: "1dba6-131b-3fd31e4a"
Accept-Ranges: bytes
Content-Length: 4891
Keep-Alive: timeout=15, max=100
Connection: Keep-Alive
Content-Type: text/html

<!DOCTYPE html>
<html>
  :
</html>
```

### リクエスト行
- リクエスト行は下記の書式で表す。
```
# リクエスト行
メソッド パス名 HTTP/バージョン
```

### レスポンス行
- レスポンス行は下記の書式で表す。
```
# レスポンス行
HTTP/バージョン ステータス番号 補足メッセージ
```

### HTTPのメソッド
- HTTPで「何をしたいか」をサーバに伝えるための命令。ファイルを読むのか？作るのか？更新するのか？削除するのか？その意図をサーバに伝えるのがHTTPメソッド。

### GET
- クライアントはサーバに対してコンテンツの取得を要求する。最も一般的に使用されるメソッド。

### POST
- クライアントはサーバに対してパラメーターを送信する。

### PUT
- クライアントはサーバに対してリソース全体書き換えを要求する。一度の変更を求める場合は PATCH、全体の変更を求める場合は PUT を使用するのが一般的。

### PATCH
- クライアントはサーバーに対してリソースの一部の書き換えを要求する。

### DELETE
- クライアントはサーバに対してリソースの削除を要求する。

### HEAD
- クライアントはサーバに対して GET と同じレスポンスを、ヘッダ情報のみ返却することを求める。

### CONNECT
- クライアントはプロキシサーバ経由で HTTPS通信を行う際、プロキシサーバにトンネリングを貼ることを要求する。

### OPTIONS
- クライアントはサーバがサポートしているメソッドの一覧の返却を要求する。

### TRACE
- HTTP要求がどのプロキシサーバを経由して送信されるかなど、HTTP の動作をトレースする際に用いる。

### LINK
- 指定した URL とリソースにリンク関係を結ぶ。

### UNLINK
- 指定した URL とリソースの間のリンク関係を解除する。
